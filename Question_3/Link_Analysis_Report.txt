==============================
Question 3 - File System and Link Analysis
==============================

1. Objective
The purpose of this experiment was to understand how Linux hard links and symbolic links work.
A regular file was created along with one hard link and one symbolic link.
Their inode numbers, metadata, and behavior were then analyzed before and after deleting the original file.


2. File Creation and Link Setup
Commands Used:
echo "This is the original file." > original.txt
ln original.txt hardlink.txt
ln -s original.txt symlink.txt

Observation:
A regular file named original.txt was created first.
Then a hard link named hardlink.txt and a symbolic link named symlink.txt were created for the original file.
This setup was used to compare how hard links and symbolic links behave in Linux.

The symbolic link symlink.txt has a different inode number (12384898975444274) because it is a separate file that stores the path of the original file.
This clearly shows that hard links and symbolic links are implemented differently in Linux.

Observation:
The cat command was used to read the contents of the original file, hard link, and symbolic link.

This shows that both the hard link and symbolic link can be used to access the original file while it still exists.
At this stage, there was no visible difference in the output returned by the three names.


6. Effect of Deleting the Original File

Observation:
After deleting original.txt, the hard link hardlink.txt continued to work and still contained the original file data.
This happened because a hard link directly references the same inode as the original file, so the data remains available as long as at least one hard link exists.
However, the symbolic link symlink.txt stopped working because it only stores the pathname of the original file.
Once original.txt was removed, the symbolic link became a broken link and could no longer access the file contents.

8. Comparison Between Hard Link and Symbolic Link
Hard Link:
- A hard link points directly to the same inode as the original file.
- It shares the same inode number as the original file.
- Deleting the original filename does not remove the file data if another hard link still exists.
- Hard links generally work within the same filesystem.

Symbolic Link:
- A symbolic link is a separate file that stores the path to another file.
- It has its own inode number.
- If the original file is deleted or moved, the symbolic link becomes broken.
- Symbolic links can point across different filesystems and can also point to directories.

Observation:
The experiment clearly showed that hard links are directly connected to the underlying file data, while symbolic links are path-based references.
The inode numbers, link counts, and deletion behavior all confirmed this distinction in practice.


9. Conclusions Drawn from the Experiment
The experiment successfully demonstrated the difference between hard links and symbolic links in Linux.
The original file and hard link shared the same inode number, confirming that both names referred to the same underlying file data.
The symbolic link had a separate inode and acted only as a reference to the original file path.
After deleting the original file, the hard link still preserved access to the file contents, whereas the symbolic link failed because its target no longer existed.
This shows that hard links are inode-based references while symbolic links are pathname-based references.
