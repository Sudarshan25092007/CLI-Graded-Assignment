5. Mounted File System Details
Command Used:
findmnt

Output Summary:
The findmnt command displayed the mount hierarchy of the WSL environment, including the Linux root filesystem, WSL-specific overlay mounts, temporary filesystems, runtime mounts, and the Windows C: drive mounted at /mnt/c.
Important mount points observed include:
- / mounted from /dev/sdd with ext4 filesystem
- /mnt/c mounted from C:\ using a 9p-based WSL integration layer
- /run, /dev, /proc, and /sys mounted as temporary or virtual Linux system filesystems
- WSL-specific mounts such as /mnt/wsl, /mnt/wslg, and /usr/lib/wsl

Observation:
The findmnt output shows that this environment is a WSL-based Linux system layered on top of Windows.
The root filesystem uses ext4, which behaves like a normal Linux filesystem for most system operations.
The Windows drive is mounted separately under /mnt/c and is integrated into Linux through the WSL mount system.
Understanding these mount points is important because storage behavior, permissions, and inode reporting may differ between the native Linux filesystem and Windows-mounted paths.

7. Storage Health Observations
The storage assessment shows that the main Linux root filesystem has very healthy capacity and inode availability.
The root filesystem mounted on /dev/sdd has approximately 954 GB free and only 1% inode utilization, which means it is well suited for application deployment and development work.
The most significant storage concern is the mounted Windows C: drive at /mnt/c, which is already at 95% usage with only around 26 GB of free space remaining.
If large project files, datasets, logs, or application artifacts are stored on /mnt/c, storage pressure may increase further.
From a Linux storage perspective, the WSL root filesystem is in good condition, but the Windows-mounted storage should be monitored carefully.


8. Recommendations for Improving Storage Utilization
1. Reduce usage on the Windows C: drive mounted at /mnt/c by removing unnecessary files, temporary folders, old downloads, and unused project artifacts.
2. Prefer storing Linux-specific experiments, logs, and temporary project data inside the native Linux filesystem rather than only on /mnt/c, especially when Linux permissions or filesystem behavior matter.
3. Regularly monitor disk usage using df -h and directory usage using du -sh or du -h to identify large folders before storage becomes critical.
4. Keep an eye on large development assets such as node_modules folders, build outputs, screenshots, and cached files, as these can quickly consume space.
5. Maintain enough free space on the Windows drive to avoid performance issues and ensure WSL and development tools continue to function smoothly.


9. Conclusion
The storage assessment successfully identified the available storage devices, mounted file systems, disk usage, inode usage, and overall storage health of the Linux environment.
The main Linux root filesystem is in a strong state with large free capacity and very low inode usage, making it suitable for application deployment.
However, the mounted Windows C: drive is heavily utilized and should be cleaned or monitored if it is used for storing development files and project data.
Overall, the Linux environment appears healthy, but storage management on the Windows side remains important for long-term stability and efficient usage.
